41 """
42 `TransparentColor{C,T,N}` is the abstract type for any
43 color-with-transparency. The `C` parameter refers to the type of the
44 pure color (without transparency) and can be extracted with
45 `color_type`. `T` is the element type of both `C` and the `alpha`
46 channel, and `N` has the same meaning as in `Colorant` (and is 1 larger
47 than the corresponding color type).
48
49 All transparent types should support two modes of construction:
50
51 P(color, alpha)
52 P(component1, component2, component3, alpha) (assuming a 3-component color)
53
54 For a `Colorant` `c`, the color component can be extracted with
55 `color(c)`, and the alpha channel with `alpha(c)`. Note that types
56 such as `ARGB32` do not have a field named `alpha`.
57
58 Most concrete types, like `RGB`, have both `ARGB` and `RGBA`
59 transparent analogs. These two indicate different internal storage
60 order (see `AlphaColor` and `ColorAlpha`, and the `alphacolor` and
61 `coloralpha` functions).
62 """
63 abstract type TransparentColor{C<:Color,T,N} <: Colorant{T,N} end

288 """
289 `RGB24` uses a `UInt32` representation of color, 0xAARRGGBB, where
290 R=red, G=green, B=blue and A is irrelevant. This format is often used
291 by external libraries such as Cairo.
292
293 `RGB24` colors do not have fields named `r`, `g`, `b`, but you can
294 still extract the individual components with `red(c)`, `green(c)`,
295 `blue(c)`. You can construct them directly from a `UInt32`, or as
296 `RGB(r, g, b)`.
297 """
298 struct RGB24 <: AbstractRGB{N0f8}
299 color::UInt32
300
301 RGB24(col::UInt32, ::Type{Val{true}}) = new(col)
302 end
303 # The Val{true} constructor solves a consistency problem, one that's
304 # most obvious for Gray24: Gray24(x) interprets x as a value between 0
305 # and 1; since a UInt32 can have value 0 or 1 (among other
306 # possibilities), having Gray24(x::UInt32) mean something entirely
307 # different (interpreting the UInt32 as a bit pattern rather than a
308 # value) would be inconsistent, confusing, and error-prone. Because we
309 # can construct an RGB24 from a grayscale value (just repeating the
310 # value for all 3 color channels), the same problem extends to all the
311 # "24/32" color types.
312
313 # However, some constructors "build" RGB24s as a UInt32 bit pattern,
314 # so we need some way of turning a UInt32 directly into an RGB24.
315 # This is the role of the Val{true} constructor. This is an internal
316 # implementation detail, and user code should not use it directly:
317 # use reinterpret(RGB24, x::UInt32) instead. (The Val{true}
318 # constructor is used to implement reinterpret, see traits.jl.)
319 function RGB24(r::N0f8, g::N0f8, b::N0f8)
320 r32 = UInt32(reinterpret(r)) << 16
321 g32 = UInt32(reinterpret(g)) << 8
322 b32 = UInt32(reinterpret(b))
323 reinterpret(RGB24, r32 | g32 | b32)
324 end

327 `ARGB32` uses a `UInt32` representation of color, 0xAARRGGBB, where
328 R=red, G=green, B=blue and A is the alpha channel. This format is
329 often used by external libraries such as Cairo. On a little-endian
330 machine, this type has the exact same storage format as `BGRA{N0f8}`.
331
332 `ARGB32` colors do not have fields named `alpha`, `r`, `g`, `b`, but
333 you can still extract the individual components with `alpha(c)`,

340 ARGB32(col::UInt32, ::Type{Val{true}}) = new(col)

342 function ARGB32(r::N0f8, g::N0f8, b::N0f8, alpha::N0f8 = N0f8(1))
343 a32 = UInt32(reinterpret(alpha)) << 24
344 r32 = UInt32(reinterpret(r)) << 16
345 g32 = UInt32(reinterpret(g)) << 8
346 b32 = UInt32(reinterpret(b))
347 reinterpret(ARGB32, a32 | r32 | g32 | b32)
348 end
